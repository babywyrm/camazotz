# The Feedback Loop — Scan, Recommend, Enforce, Validate

This guide walks through the complete cycle of finding vulnerabilities,
generating defenses, applying them, and proving they work.

---

## The Loop

```
1. SCAN        mcpnuke finds vulnerabilities in the MCP server
                  ↓
2. RECOMMEND   mcpnuke --generate-policy produces nullfield YAML
                  ↓
3. ENFORCE     kubectl apply the policy → nullfield hot-reloads
                  ↓
4. VALIDATE    Re-scan to confirm findings are gone, or run defense labs
                  ↓
               Loop back to 1 for regression testing
```

---

## Step 1: Scan

Run mcpnuke against camazotz (locally or on the cluster):

```bash
mcpnuke --targets http://localhost:8080/mcp --fast --verbose
```

Example output (abbreviated):


## Step 2: Generate Policy

Use `--generate-policy` to produce a nullfield policy from the findings:

```bash
mcpnuke --targets http://localhost:8080/mcp --fast --generate-policy fix.yaml
```

This produces a ready-to-apply NullfieldPolicy:

```yaml
# Generated by mcpnuke --generate-policy
apiVersion: nullfield.io/v1alpha1
kind: NullfieldPolicy
metadata:
  name: mcpnuke-recommended
spec:
  selector:
    matchLabels: {}
  rules:
    - action: DENY
      mcpMethod: tools/call
      toolNames: ["shadow.register_webhook"]
      reason: "mcpnuke: webhook persistence vector"
    - action: HOLD
      mcpMethod: tools/call
      toolNames: ["hallucination.execute_plan"]
      reason: "mcpnuke: code execution capability"
      hold:
        timeout: "5m"
        onTimeout: DENY
    - action: SCOPE
      mcpMethod: tools/call
      toolNames: ["relay.execute_with_context"]
      reason: "mcpnuke: credential parameter exposure"
      scope:
        response:
          redactPatterns: ["password", "secret", "token", "api_key", "credential"]
    - action: DENY
      mcpMethod: tools/call
      toolNames: ["*"]
      reason: "mcpnuke: default deny"
```

The mapping logic:
- **DENY** for webhook persistence and exfiltration vectors
- **HOLD** for code execution (require human approval)
- **SCOPE** for credential exposure (redact sensitive response content)
- **BUDGET** for rate limit findings (per-identity/session call caps)
- **Default DENY** at the bottom catches anything not explicitly allowed

## Step 3: Apply the Policy

### Local (Docker Compose)

Mount the generated policy into the nullfield sidecar:

```bash
docker compose -f docker-compose.yaml up -d
# nullfield reads /etc/nullfield/policy.yaml — mount fix.yaml there
```

### Kubernetes (CRD)

Apply as a Custom Resource:

```bash
kubectl apply -f deploy/crds/  # install CRDs (one-time)
kubectl apply -f fix.yaml      # apply the generated policy
```

The nullfield controller watches for CRD changes and syncs to ConfigMaps.
The sidecar's hot-reload detects the ConfigMap update and swaps the policy
engine within seconds — no pod restart needed.

### Kubernetes (ConfigMap)

Or apply directly as a ConfigMap:

```bash
kubectl -n camazotz create configmap nullfield-policy \
  --from-file=policy.yaml=fix.yaml --dry-run=client -o yaml | kubectl apply -f -
```

---

## Step 4: Validate

### Option A: Re-scan

Run mcpnuke again. Findings that are now blocked by nullfield should
produce different results — tool calls that previously succeeded now
return `-32000` (policy denied) or `-32005` (hold timeout).

```bash
mcpnuke --targets http://localhost:8080/mcp --fast --verbose
# Expect: fewer findings, blocked attack chains
```

### Option B: Defense labs

Use the camazotz defense-mode labs to practice writing policy manually:

| Lab | What You Practice |
|-----|------------------|
| `policy_authoring` | Write DENY/HOLD/SCOPE rules for a known attack chain |
| `response_inspection` | Craft SCOPE redactPatterns for credential leaks |
| `budget_tuning` | Set BUDGET limits for mixed legitimate + malicious traffic |

These labs score your policy on effectiveness (does it block the attack?)
and precision (does it allow legitimate use?).

### Option C: Save baseline

Save the current scan as a baseline for regression testing:

```bash
mcpnuke --targets http://localhost:8080/mcp --save-baseline baseline.json
```

On future scans, compare against the baseline:

```bash
mcpnuke --targets http://localhost:8080/mcp --baseline baseline.json
```

New findings that weren't in the baseline are flagged as regressions.

## Putting It All Together

The complete workflow for a security team adopting this stack:

1. **Deploy camazotz** as the vulnerable MCP target (staging/lab environment)
2. **Run mcpnuke** to inventory the attack surface
3. **Generate policy** with `--generate-policy` as a starting point
4. **Review and refine** the policy — the generated YAML is a recommendation,
   not a final answer. Adjust actions, add ALLOW rules for tools you need,
   tune BUDGET limits for your traffic patterns.
5. **Apply to nullfield** — sidecar or gateway mode
6. **Re-scan** to confirm the policy blocks the findings
7. **Run defense labs** to train your team on policy authoring
8. **Save baseline** for CI/CD regression testing
9. **Repeat** as you add new tools or the threat landscape changes

The three projects reinforce each other:
- **camazotz** provides the realistic vulnerable target
- **mcpnuke** finds the vulnerabilities and generates the fix
- **nullfield** enforces the fix and proves it works

No single tool solves the problem. The loop does.
